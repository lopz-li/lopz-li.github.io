初次体验特斯拉Model 3

发表于 2022-07-11 更新于 2023-04-24 分类于 记录 ， 特斯拉 阅读次数：

上周租了两天特斯拉Model 3开了开，属于是第一次长时间体验了，记录一下。

车型

这次租的是2020款的最普通那款Model 3，携程下单，收费上门送取车。总体感觉服务不错，交还车都很顺利，车子也比较干净，车况很好。

驾驶

  • 加速

电车的加速响应确实很快，可以说是随心所欲的感觉，吊打各种自动挡。空旷的路上试了下加速能力，没踩到底，但感觉已经非常猛了。路上各种超车毫无压力。

  • 单踏板

特斯拉争论最多的就是所谓的单踏板模式，其实就是强能量回收加上Auto Hold。由于能量回收力度很强，并且跟加速踏板深度有关，速度低至临近停止时，车辆会自动控制刹车刹死。这样大部分情况下，驾驶者只需要控制加速踏板即可，“单踏板模式”由此得名。实际上两天开下来，感觉这种驾驶方式还是很容易适应的，松加速踏板减速的特性只需要几分钟就适应了，路上遇到比较紧急的情况时，右脚在加减速踏板之间的切换也都是下意识的，没有网上说的那么多问题。在山路下坡等一些情况下，能量回收的刹车力度也会有不足的现象，需要踩刹车进一步减速。全程下来觉得一个踏板搞定的这种方式一定程度上还是比普通油车要省劲的。

  • 辅助驾驶

高速和快速路上尝试了特斯拉标配的辅助驾驶AP功能。刚开始用的时候还是有些怕的，对它的能力边界没什么底，适应了一段时间之后就比较放心了。租车的两天时间内基本都是晴天，路面视野非常好，特斯拉能很好地识别车道线并保持在车道中间。遇到前方有车的时候，能自动减速，此时可先打转向灯，然后手动控制超车，车道保持会失效，但速度保持扔继续工作，后方无车干扰时，车会自动加速，驾驶者只需要控制好方向即可。超车完成后，需要再次手动启动车道保持功能。

开启辅助驾驶时如果觉得车速较慢，可以踩加速踏板加速，这种情况下辅助驾驶系统遇到前方障碍时不会自动刹车，实际体验时，这种情况下能量回收模式貌似和不开辅助驾驶时不太一样，加速踏板的脚感和正常驾驶时也有所不同，需要十分注意。

车身不在车道正中时如果开启辅助驾驶，在靠边的车道上会有一股很大的力量将方向盘拉到撞向车道外边线的方向，让人觉得车子要甩出车道，但车身在车道正中的时候开启则不会有这种现象，需要十分注意。

开启辅助驾驶一段时间之后，由于注意力逐渐转移，亲身体验驾驶者的安全意识会有一定程度的松懈，看手机之类的小动作持续时间会显著延长，这一点需要非常注意！！！

充电

此次租车中间一共充了四次电，分别是一次景区的慢充、两次高速服务区快充、一次商业体停车场快充，总体感受还是不错的。景区停车场提供了很多慢充桩，功率7kW，绝大部分可用，进去玩时插上，出来时可以充不少里程，挺方便的。高速服务区貌似都是电网公司的快充桩，会有部分桩不支持特斯拉、小鹏等一些车型，充电速度一般，峰值50多kW，但大部分时候只有30多kW。商业体停车场的是开迈斯快充，最高到90多kW，速度很快。

其他

空调效果不错，打开之后很快就能把温度降下来。

租车的两天太阳很大，很晒很热，但天窗玻璃感觉并不热，只是手摸上去很烫，除此之外并没有什么感觉。

噪音方面，感觉和自己的油车噪声水平区别并不是很大，噪声源主要是胎噪和风噪，和油车的发动机噪声不太一样，没有什么不舒服的感觉。

电车停着可以开空调真是香。

从Ubuntu迁移到Debian：使用NetworkManager管理网络连接

发表于 2022-07-05 更新于 2023-04-24 分类于 记录 ， Debian 阅读次数：

在物理机上安装Debian之前，我先在虚拟机上体验了一段不短的时间，发现了Ubuntu和Debian的一个小区别：Ubuntu安装好之后通过gnome系统的设置功能是可以配置网络的，而Debian安装好之后在gnome系统设置中都看不到网络连接。后来各方查找，发现需要对NetworkManager进行一些配置才能让其正确接管网络配置。

系统/etc/NetworkManager下的NetworkManager.conf配置文件中有一个[ifupdown]字段，其中有一个managed配置项，取值true时由NetworkManager管理网络连接，此时在gnome系统设置中就能看到网络连接了。

另外我在设置网卡固定IP时发现，通过gnome系统设置网络后，需要重新启动设置才能生效，不知道是估计设计成这样的还是BUG，虽然有点麻烦但还不太影响使用。

还有后来我在物理机上安装好Debian之后，发现NetworkManager的这个设置选项即使是false，在gnome系统设置中也是能看到后接入的USB有线网卡连接的…

搞得我都有些迷糊了…

从Ubuntu迁移到Debian：选择带无线网卡支持的netinst镜像

发表于 2022-07-05 更新于 2023-04-24 分类于 记录 ， Debian 阅读次数：

在2022年当下，相当数量的笔记本都已经不提供有线网络接口了，如果想以netinst镜像方式安装Debian就只能通过无线网卡联网（有USB网卡的除外），而Debian的官方netinst镜像不带non-free的firmware，很多机器上启动之后因为无线网卡不能使用而无法继续安装过程，因此需要改用unofficial的附带non-free版本firmware的镜像以便使用无线网卡。

unofficial的镜像在官方以及各镜像站都有提供下载，路径一般是类似debian-cdimage/unofficial/non-free/cd-including-firmware这种。例如我一般使用合肥中科大的镜像，就是http://mirrors.ustc.edu.cn/debian-cdimage/unofficial/non-free/cd-including-firmware/11.3.0+nonfree/amd64/iso-cd/firmware-11.3.0-amd64-netinst.iso

另外unofficial下可能在images-including-firmware这个路径下也能找到这个带firmware的镜像，通过checksum对比可以知道两个路径下的镜像文件是完全相同的，使用任意一个都行。

从firmware版本镜像启动后，可以通过ip link命令查看无线网卡是否正常识别。

滑雪日记之湖北省内滑雪场印象

发表于 2022-02-18 更新于 2023-04-24 分类于 记录 ， 运动 ， 滑雪 阅读次数：

对湖北省内部分滑雪场的一些印象

2022年元旦本来和同事约着一起到大别山滑雪，结果临行前因为学校不让娃出武汉市，同事一家去不了了。而我们两人兴趣却被勾起来了，加上时间又很自由没人管，于是提前一天出发，开始了滑雪体验之旅。

话说早在2007年还在日本的时候就滑过一次雪，虽然到最后都不怎么会，但还是觉得挺好玩的，这次开新坑感觉可能要出不来了…

以下是对21-22雪季中所到过的湖北省内的一些滑雪场的印象。

大别山南武当滑雪场

我们首先去的是位于湖北省英山县大别山南武当旅游区的南武当滑雪场，从武汉出发自驾大约3小时左右。雪场有一条初级道和一条中级道，坡度适中，雪具大厅出来是坡顶，雪道尽头是面积不大的平地区。对于见识过日本超大雪场的我来说，南武当雪场第一眼看上去是有些失望的，我们到雪场的时候恰好碰到一波旅游团到达，小小的雪场瞬间人满为患。于是当场就准备转道去传说中湖北最好的神农架滑雪场，但导航搜索发现从南武当雪场出发要8个多小时才能到…这样就又犹豫了，抱着先考虑考虑的心态到附近预订好的藏龙大别山度假酒店吃了个午饭，最终决定还是先在南武当雪场就近体验一下。

因为本人就是入门水平，老婆更是一次都没滑过，下午在南武当雪场的滑雪体验感觉还是不错的。进雪场的时候人不是很多，给老婆请了个教练教一下基本的犁式刹车，我在旁边跟着找找雪感，几圈下来慢慢找到了一点感觉。整体感受雪道坡度适中，人不多的时候作为初学者练习的场地是足够了。

设施方面，南武当初中级雪道共用唯一的一条魔毯，雪具大厅提供常规的雪服、手套、雪镜、雪鞋、雪板租赁，我们是直接现场购票，票价包含了租赁费用，储物柜需要单独收费。

住宿方面，我们选择的是以前来玩的时候住过的藏龙大别山度假酒店，离滑雪场不远，大约10分钟车程。这个酒店价格较高，纯粹是为了想再回来看看才选择的，附近是否有其他更实惠的酒店或民宿就不得而知了。

用一句话总结对南武当雪场的印象的话，就是适合初学者练习。

咸宁九宫山滑雪场

九宫山滑雪场位于咸宁市通山县，从武汉出发自驾大约3个半小时左右到达。雪场有两条雪道，雪道上部坡度较陡，下部较上部稍缓，坡度不均匀，整体感觉略陡，雪道尽头是面积很大的平地区，雪具大厅在平地区的尽头。

如果是自驾的话，只能上到九宫山云中湖附近酒店集中区域，然后需要换乘免费大巴上到滑雪场。对于去过南武当的我们，对九宫山滑雪场的第一感觉是规模稍大一些，雪道坡度偏陡。我们在雪场泡了两天半，感觉雪道维护得一般，可能是由于气温的原因，雪道上经常有冰，对初学者不是很友好。另外每天下午约1点开始就会有很多旅行团进场，大量的游客将平地区塞得满满的，雪道上下来的人如果不会控制速度就容易发生事故。九宫山雪场的教练喜欢倒滑带教，学员较多时，雪道很快就会被刮出几道沟，加上雪道上人很多，作为不熟练的初学者经常不敢下道，可以说九宫山对想练习滑雪技术的初学者是不太友好的，每天只有赶早人少的两三个小时适合练习。

设施方面，九宫山只有一条魔毯，魔毯的出入口处有台阶，需要脱板上下。雪具大厅提供雪服、手套、雪镜、雪鞋、雪板租赁，储物柜需要单独收费，雪场提供免费的鞋柜。雪具大厅二层是餐饮区，餐食质量一般，选择泡面的人很多。

住宿方面，云中湖周边有不同价位的酒店，我们待的几天中开门的酒店数量算不上很多，周边的餐饮同样如是，可能因为来长住滑雪的人并不是很多，生意不好的原因吧。

用一句话总结对九宫山雪场的印象的话，就是拍照的人超多，想练活的话要早上赶早来。

Windows环境下编译rocketmq-client-cpp（Visual Studio 2017 Community）

发表于 2021-12-25 更新于 2023-04-24 分类于 记录 ， rocketmq 阅读次数：

以为很轻松，结果是个大坑！！！

以前帮同事编译过Linux环境下的rocketmq-client-cpp库，用源码自带的构建脚本非常顺利，过程中需要从github下载第三方依赖库的代码，速度会比较慢，但加上代理之后很快就能完成了，整个过程没有费什么工夫。

这次又要编一个Windows下的版本，因为也有Windows环境下的构建脚本，以为也会非常轻松，结果没想到跳了个大坑！

所幸一波三折之后终于是成功了，功能还有待测试，这里记录一下编译过程。

依赖库下载

官方readme列举了几个第三方依赖库

  • boost 1.58.0
  • jsoncpp 0.10.6
  • libevent 2.0.22
  • zlib 1.2.3

其实还有一个没有列出的

  • openssl 1.0.2

以上依赖库可以自己下载源码之后，解压到rocketmq-client-cpp目录下的thirdparty目录下（没有可以自行新建），保留原始目录名（需要删去结尾的-master后缀）以省去一些麻烦。自动构建脚本也有通过github直接clone，但没代理的话速度比较慢。

构建环境

构建脚本是基于Visual Studio 2015编写的，我的环境是Visual Studio 2017 Community，需要额外安装一下Visual Studio 2015的支持。

需要注意的是安装好Visual Studio 2015后生成的命令行提示符环境内路径设置有问题，不能执行devenv程序，需要从Visual Studio 2017命令行提示符进入之后，再执行Visual Studio 2015的环境变量设置脚本设置编译环境

依赖库构建

boost

编译boost可参考自动构建脚本中的步骤进行，需要注意的是自动构建脚本编译时没有加入zlib支持，后续编译rocketmq-client-cpp时会报错，此处需要自行添加zlib支持。

其中-sZLIB_SOURCE=\path-to-zlib-source指定zlib源文件路径，-SZLIB_INCLUDE=\path-to-zlib-source指定zlib头文件路径，均设置为解压后的zlib库根路径即可，路径分隔符采用windows下的\，设置为绝对路径可以避免一些不必要的麻烦

jsoncpp

编译jsoncpp比较简单，参考自动构建脚本即可

libevent

编译libevent可参考自动构建脚本，需要注意的是libevent源码中的Visual Studio工程文件没有添加openssl支持，需要手工添加bufferevent_openssl.c源文件，并在工程配置中添加_EVENT_HAVE_OPENSSL宏定义以及openssl头文件路径

另外编译过程中可能会出现找不到Winsock2.h等头文件的错误，需要手工添加头文件路径，根据机器环境头文件目录各不相同。我的是C:\Program Files (x86)\Windows Kits\10\Include\10.0.17134.0\shared和C:\Program Files (x86)\Windows Kits\10\Include\10.0.17134.0\um这两个

工程文件准备好之后即可进行构建

构建rocketmq-client-cpp

rocketmq-client-cpp的Visual Studio项目文件问题很多，编译中会出现很多错误，以下记录了对工程文件的一些修改

  1. 项目属性中的默认平台工具集会导致编译不过，需要修改为Visual Studio 2015
  2. 可能出现找不到Winsock2.h等头文件的提示，需要手工添加头文件路径，例如我的就是C:\Program Files (x86)\Windows Kits\10\Include\10.0.17134.0\shared和C:\Program Files (x86)\Windows Kits\10\Include\10.0.17134.0\um这两个
  3. 会报找不到MQClient.cpp这个文件，项目中并没有这个文件，从工程中删除这个文件记录即可
  4. 编译中会出现很多看起来和socket及网络相关的宏、结构体的重定义报错，需要修改src/transport/SocketUtil.h和include/MQMessageExt.h两个包含了Winsock2.h和Windows.h头文件的源文件，将包含Winsock2.h的行放到Windows.h之前
  5. 编译中会出现找不到一些库自身的以及依赖库的头文件，原因是没有添加到头文件搜索路径中，需要手工添加，主要有src/trace、src/include、openssl库头文件目录
  6. 编译中部分源文件会报boost::weak_ptr未定义错误，原因是没有包含对应头文件，需要手工修改源文件，增加#include <boost/weak_ptr.hpp>项
  7. StringIdMaker.cpp会报找不到重载函数的错误，原因是函数定义和函数头文件声明的签名不同，将函数定义处的签名改成和头文件一致即可
  8. TcpTransport.cpp会报无法从void转换到const char的错误，将源文件中的void修改为char即可
  9. TraceContant.h会报意外的标号错误，原因是静态变量名被定义了同名宏，需要手工取消宏定义
  10. EventLoop.cpp会报找不到bufferevent_openssl_socket_new符号的错误，原因是没有包含对应头文件，需要手工修改源文件，添加#include <event2/bufferevent_ssl.h>项
  11. 链接过程中会报很多无法解析外部符号错误，原因有两种，一部分找不到的符号是因为没有引用依赖库文件，需要在工程设置中手工添加依赖库文件引用以及库文件搜索路径；另一部分找不到的符号是因为rocketmq-client-cpp的工程文件有错误，没有添加一些源文件，根据错误信息中的类名等可搜索同名源文件所在位置，添加到工程中之后再次构建即可

感想

玩C/C++也有很多年了，它的构建系统确实是比其他语言麻烦点，但如果是我自己的工程，可能不能说拿到哪都直接能用，却也不至于到像rocketmq-client-cpp这种完全没法用的程度。

原以为既然是阿里这种大公司的开源项目，肯定都是年薪百万的程序员搞出来的，应该非常方便好用，折腾了这一遭看来也不过如此啊！

但是回想一下，在Linux环境下确实还是很顺利的，莫不是阿里对Windows已经放弃治疗了？

再次运行，成功！！！

这样每次运行都要打命令有点麻烦，于是准备写一个shell，每次双击就可以了，这下跳坑了…

直接写脚本，加上自动补全绝对路径之后，wine总是报各种找不到exe文件，完全摸不着头脑。

经过一白天的研究，总结了以下几条

  1. 需要设置WINEPREFIX环境变量，指向wine的配置环境，一般都是/home/username/.wine这样（就是用户主目录下的.wine，winecfg配置好的环境一般都在这里）
  2. 没有任何指定的情况下，wine默认会在配置环境目录下去找exe文件，所以要么cd到exe文件所在路径后再执行wine，要么在wine参数中带exe文件的完整路径

确定了原则之后，重新写了脚本，执行一次成功。
